# XiangShan RepCut SimTop 分区端口连接与位宽统计（2026-03-26）

## 1. 范围与方法

统计对象只看 [xs_wolf_repcut.json](/workspace/wolvrix-playground/build/xs/repcut/xs_wolf_repcut.json) 中 `graphs[*].symbol = SimTop_repcut_part*` 的 32 个分区，不分析 graph 内部节点。

这次统计同时使用：

- `ports.in`
- `ports.out`
- `vals[*].w`

因此既统计“连接个数”，也统计“传输 bit 量”。

统计口径如下：

- 若某个 `val` 出现在分区 A 的 `ports.out`，同时出现在分区 B 的 `ports.in`，则记为一条 `A -> B` 的跨分区连接。
- 若某个 `val` 只出现在 `ports.in`，则视为外部输入值。
- 若某个 `val` 只出现在 `ports.out`，则视为外部输出值。
- 同一个 `val` 可能扇出到多个目的分区，因此同时统计：
  - `signal_count`：唯一 value 数量
  - `port_count`：实际投递次数；若一个 value 扇出到 3 个目的分区，则计 3 次
  - `signal_bits`：每个唯一跨分区 value 的位宽只记 1 次
  - `port_bits`：按实际投递次数累计位宽；若一个 64bit value 扇出到 3 个目的分区，则计 `192 bit`

补充说明：

- 当前 SimTop 分区对内没有发现“同一个 `src -> dst` 对内重复消费同一 value”的情况，因此对单个 `src -> dst` 分区对而言，`signal_count = port_count`，`signal_bits = port_bits`。
- 之前文档只看 `ports.in/out` 做了首轮统计；这次做 width-aware 复扫时顺带校正了少量端口总数误差。以下数字以本次复扫为准。

## 2. 整体规模

从 SimTop 分区接口看，当前 RepCut 的跨分区连接非常密集，而且外部输入输出极少，绝大多数接口值都在 32 个分区之间流动。

| 指标 | 数值 |
| --- | ---: |
| 分区数量 | `32` |
| 输入端口总数 | `575037` |
| 输出端口总数 | `217488` |
| 有 producer 的唯一输出 value 数 | `217488` |
| 有 consumer 的唯一输入 value 数 | `217490` |
| 跨分区 value 数 | `217483` |
| 外部输入 value 数 | `7` |
| 外部输出 value 数 | `5` |
| 有连接的有向分区对数量 | `798` |
| 有向分区对理论上限 | `992 = 32 * 31` |
| 分区对连通密度 | `80.44%` |
| 平均每个 value 扇出到的目的分区数 | `2.64` |
| 扇出目的分区数中位数 | `1` |
| 最大扇出目的分区数 | `31` |
| 唯一跨分区 payload 总量 `signal_bits` | `2520097 bit` |
| 实际投递总量 `port_bits` | `5039626 bit` |
| 位宽平均值 | `11.59 bit/value` |
| 位宽中位数 | `2 bit` |

几个直接结论：

- 从 value 个数看，绝大多数跨分区值都很窄，中位数只有 `2 bit`。
- 但从 bit 总量看，真实 payload 并不小，唯一跨分区值累计已经有 `2520097 bit`。
- 考虑扇出之后，实际 update 推送总量膨胀到 `5039626 bit`，约等于唯一 payload 的 `2.00x`。
- 外部输入只有 `7` 个 value，外部输出只有 `5` 个 value，说明 SimTop 分区接口几乎完全是内部互联。

## 3. 位宽分布

如果只看 value 数量，会误以为系统几乎全是窄信号；但按 bit 数重新看，结论明显不同。

| 位宽桶 | `signal_count` | value 占比 | `signal_bits` | bit 占比 | 平均位宽 |
| --- | ---: | ---: | ---: | ---: | ---: |
| `1` | `87211` | `40.10%` | `87211` | `3.46%` | `1.00` |
| `2-8` | `89930` | `41.35%` | `325455` | `12.91%` | `3.62` |
| `9-32` | `10991` | `5.05%` | `149245` | `5.92%` | `13.58` |
| `33-64` | `28165` | `12.95%` | `1708688` | `67.80%` | `60.67` |
| `65-128` | `625` | `0.29%` | `70705` | `2.81%` | `113.13` |
| `129-256` | `357` | `0.16%` | `88173` | `3.50%` | `246.98` |
| `257+` | `204` | `0.09%` | `90620` | `3.60%` | `444.22` |

最重要的几个观察：

- `1bit` 和 `2-8bit` 信号合计占了 `81.45%` 的跨分区 value 数，但只占 `16.37%` 的唯一 payload bit。
- `33-64bit` 信号只占 `12.95%` 的 value 数，却占了 `67.80%` 的唯一 payload bit，是绝对主导的 payload 层。
- `257+` 的超宽值数量非常少，但仍贡献了 `3.60%` 的 bit，总体上不能忽略。

## 7. 接口形态判断

从 SimTop 分区接口看，当前连接形态有几个很鲜明的特点：

1. 外部接口极少，几乎全部接口值都在分区之间流动。
2. 图整体高度连通，`798 / 992 = 80.44%` 的有向分区对都有连接，不像简单的链式流水。
3. 从 value 数量看，大量连接是窄信号、点对点传输。
4. 从 payload bit 量看，真正主导的是 `33-64bit` 这层中宽总线。
5. 广播放大量接近 `2.00x`，说明 update phase 的压力不只是“值有多宽”，还包括“被投递到多少个目的分区”。

## 8. 对 runtime 的含义

这份 width-aware 统计对当前 `global_update / update_push` 模型有几个直接含义：

1. 如果只看 value 数量，优化目标会偏向 `part24`、`part11`、`part23`。
2. 如果再把位宽加进去，必须把 `part29`、`part25`、`part26`、`part13` 这类“宽总线热点”一起纳入。
3. `part26` 虽然不是最大的 count sink，但它是最大的 bit sink，后续分析 update 慢分区时不能漏掉它。
4. `part24 -> part13`、`part29 -> part26`、`part25 -> part26`、`part13 -> part26` 这几条通路，是后续定位 update push 长尾时最值得优先对齐 timing 的候选对象。
5. 由于当前没有发现同一 `src -> dst` 对内重复消费同一 value 的情况，优化重点应更偏向：
   - 跨目的分区广播放大
   - 宽总线通路的写入局部性
   - 重 source 分区的任务切分与负载均衡
